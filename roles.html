ARIA Landmark Role Summary

Role Attributes should only be placed on an HTML5 equivalant element (nav gets a role of navigation etc) or non-semantic elements because they do not have semantic meaning. If you place a role on a semantic element (like a ul) most screen readers will override the role with the element's native one with the sepecified one.

Banner

HTML5 Equivalant to <header>

Aria Banner Role is for site-wide sections of content, and to be added to either a header or div, it can be added to other elements but semantics and best practice advice against. There should only be one banner role per page of content. Header elements are not considered landmarks when they are contained within a section or article element.

See the Pen role="banner" by Rebekah Padula (@rebekahpadula) on CodePen.

Navigation

HTML5 Equivalant to <nav>

Use a nav or div as the container of a navigation list that is created with a ul or ol tag.

The container's role should be set to navigation

For pages/sites with more than one navigation present at a time, aria-label is required on the container to give a description of each navigation's purpose. If you want to give more information on a navigation you can use aria-describedby but this is not required.

See the Pen Navigation by Sara Kennedy (@slkennedy) on CodePen.

Complementary

HTML5 Equivalant to <aside>

Complementary roles are the equvilant of using an aside. Think of them as tertiary content just the same as an aside as you add roles to your code.

A suppoerting section of the document, designed to be complementary to the main content at a similar level in the DOM hierarchy, but remains meaninful when seperated from the main content. The complementary role indicated that contained content is relevant to the main content.

Some examples might be show times, current weatherm related articles, or stocks to watch.

See the Pen role="complementary" by Rebekah Padula (@rebekahpadula) on CodePen.

Contentinfo

HTML5 Equivalant to <footer>

The contentinfo role is usually for the footer of the page. Again, one per page. Footer elements are not considered landmarks when they are contained within a section or article element.

See the Pen role="contentinfo" by Rebekah Padula (@rebekahpadula) on CodePen.

Form

HTML5 Equivalant to <form>

The form role contains form elements which can be submitted by user. Please note that this is considered a navigable landmark only when labelled with aria-label or aria-labelledby.

A form may be a mix of host language form controls, scripted controls, and hyperlinks. Authors are reminded to use native host language semantics to create form controls whenever possible. For search facilities, authors SHOULD use the search role and not the generic form role. Authors SHOULD provide a visible label for the form referenced with aria-labelledby. If and author uses a script to submit a form based on a user action that would otherwise not trigger an onsubmit event, the author SHOULD provide the user with advance notification of the behavior. Use native host language sematics to create form controls whenever possible.

When testing the form role I found that a div containing form elements (inputs) does not throw any accessibility errors like I would expect.... My question is will the form tag be treated/announced appropriately by screen readers without the aria role?

See the Pen role="form" by Rebekah Padula (@rebekahpadula) on CodePen.

Log

HTML5 Equivalant to <form>

A type of live region where new information is added and old information may disappear. Unlike other live regions, in this role there is a relationship between the arrival of new items in the log and the reading order.

New information is added only to the end of the log, not at arbitrary points.

Examples include chat logs, message history, game log, or an error log.

Note that elements with the role log have an implicit aria-live value of polite but you can change it if you want.

See the Pen role="log" by Rebekah Padula (@rebekahpadula) on CodePen.

Main

HTML5 Equivalant to <main>

The main role is for the main content of the page. Therer should only be one main landmark per page.

See the Pen role="main" by Rebekah Padula (@rebekahpadula) on CodePen.

Region

HTML5 Equivalant to <section>

                    <div class="support" role="region">
                    </div>

NOTE THIS IS NOT AN ARIA LANDMARK

This is also a generic landmark. It should only be used when a more specific role is not available. Assistive technologies should treat this role as a complimentary region. Please note that this is considered a navigable landmark only when labelled with aria-label or aria-labelledby.

Article

HTML5 Equivalant to <article>

                    <article class="support" role="article">
                    </article>

NOTE THIS IS NOT AN ARIA LANDMARK

Search

HTML5 Equivalant to <form>

                    <main class="primaryContent" role="search">
                        <label for="search">Search Here</label>
                        <input type="text" id="search" role="" />
                    </form>

Reserved for the search tool.

status

HTML5 Equivalant to <form>

                    <form class="form" role="status">
                    </form>

Container for advisory info or feedback.

application

HTML5 Equivalant to <div>

                    <div class="primaryContent" role="application">
                    </div>

Represents a unique software unit. Keyboard commands handled by application rather than browser itself. Use sparingly if at all. The times it is necessary to be used will be few and far between for the most part. Please note that this is considered a navigable landmark only when labelled with aria-label or aria-labelledby.

presentation

                    <ul role="presentation">
                        <li>Item 1</li>
                        <li>Item 2</li>
                    </ul>

This removes any semantic meaning from the element to which it is assigned. This also applies to elements' children that are required (lists or tables) the child items will inherit the role from the parent.

using this label tells assistive technology that the element is semantically neutral and irrelevant for accessibility.

Do not use this if the element is focusable.

Do not use this if it has any ARIA attribute other than hidden.

Do not use if the element is referenced by another elements ARIA controls or properties.

Do not use if a required child element has been explicitly marked with a role (ex role="button") the child will have a role of 'button' and not 'presentation'
