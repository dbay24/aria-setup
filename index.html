ARIA General Guidelines & Rules

ARIA General Use Rules

  • If you can use native HTML elements or attributes with the semantics and behavior you need built in, do that instead of repurposing an element and adding an ARIA role, state or property.
  • Do not change native semantics unless you really have to.
  • All ARIA controls must be useable by keyboard.
  • Do not use role="presentation" or aria-hidden="true" on any visible focusable element.
  • All interactive elements must have an accessible name.

ARIA Hints & Best Practices

  • Do not disable native browser focus styles
  • Keep link underlines
  • Add skip links at the top of the page in order to guide the user over areas they would like to skip
  • Do not break heading order
  • Hide content when it isn't in focus. Use aria-hidden="true" on wrapper div, everything in that div is hidden until that property is changed
  • Break your page into percievable block areas which contain semantically associated information called "regions":
    • Assign landmark roles to your regions
    • Each region should have a percievable header used to label the region
    • Assign semantic navigation landmarks to the region.
  • Be careful when you chose to auto-focus from field to field. In the event you choose to do that you must still give user firm control over when to use it because they may not be done with a field and it is a user frustration
  • Alert your user when a max-width has been reached in an input field
  • Do not set anything to auto-submit
  • Ensure color is not the primary means of communication

ARIA Keyboard Requirements

This is a requirement, not just a best practice section.

Keyboard Bindings/Behaviors

  • We must support users who cannot use a pointing device
  • tabindex attribute is applicable to all renderable html elements:
    • 'tab' & 'tab + shift' move focus among widgets and standard html controls
    • element with tabindex = 0 is added to tab sequence based on document order.
    • element with tabindex > 0 will be added to tab sequence based on that # value
    • element with tabindex < 0 will not be added to the tab sequence, but are enabled to recieve keyboard focus.
  • Once a widget has focus, arrow keys, spacebar, enter or other keys should be able to be used to navigate the options of the widget, change its state, or trigger its function(submit etc)
  • Use onfocus to track the current focus.There is no standard DOM interface to get current focus, so you'll need to use JS to keep track of it.
  • Follow keydowns to move focus. a keydown event handler can determine the next object to recieve focus and call that element's method. Use onkeydown, not onkeypress.
  • refresh the element recieving focus's style using onfocus() to avoid inconsistent browser experiences.
  • Disabled controls should not be in the tab order.
  • Dont use :focus or attribute selectors to style the focus. IE issue. Instead, set the style in an onfocus event handler.
  • Always draw focus for tabindex="-1" or any items that recieve focus IE will not always draw focus on items that are getting focus programatically.
  • Prevent used key events from performing browser functions whilst in widget. For instance, if the arrow key is pressed, prevent it from scrolling the page with some JS code.
  • If you need to have tooltips in your code (things that on hover pop up with additional info) there must be a way to activate it or access it via keyboard and with assitive technology. Use HTML title attribute as often as possible, as assistive technology will read/render it for tooltips. Currently you can also use JS if you need to to show/hide content based on focus.
