ARIA Landmark Summary

Role Attributes should only be placed on div or span elements b/c they do not have semantic meaning. If you place a role on a semantic element (like a ul) most screen readers will override the role with the element's native one with the sepecified one.

Banner

                    <div role="banner">
                        <div class="alert">
                            <h1 class="alert__heading">
                               Warning!
                            </h1>
                            <section class="alert__message">
                                At 5:30 EST, severe weather will be incoming!
                            </section>
                        </div>
                    </div>

Aria Banner Role is for site-wide sections of content, and to be added to either a header or div, it can be added to other elements but semantics and best practice advice against. There should only be one banner role per page of content.

Navigation

                    <nav class="primaryNav" role="navigation">
                        <div class="nav__list" role="">
                            <span role=""><a class="nav__item" href="">Logout</a></span>
                            <span role=""><a class="nav__item" href="">Home</a></span>
                            <span role=""><a class="nav__item" href="">About</a></span>
                            <span role=""><a class="nav__item" href="">Contact</a></span>
                            <span role=""><a class="nav__item" href="">Locations</a></span>
                        </div>
                    </nav>

One caveat, using lists makes it more difficult for people using screen readers as it is not conducive to the logic for it. It has been suggested to replace unordered lists with spans and divs, as these are invisible to screen readers.

Primary Layout Content return of the Jedi

The main content section of the site should have the role="main" applied to it.

The Tab key should provide focus to the widget as a whole. For example, tabbing to a menu bar should put focus on the menu's first elem. The arrow keys should allow for selection or navigation within the widget. For example, using the left and right arrow keys should move focus to the previous and next menu items. When the widget is not inside a form, both the Enter and Spacebar keys should select or activate the control. Within a form, the Spacebar key should select or activate the control, while the Enter key should submit the form's default action. If in doubt, mimic the standard desktop behavior of the control you are creating.
