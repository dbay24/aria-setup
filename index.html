ARIA General Guidelines & Rules

ARIA General Use Rules

  • If you can use native HTML elements or attributes with the semantics and behavior you need built in, do that instead of repurposing an element and adding an ARIA role, state or property.
  • Do not change native semantics unless you really have to.
  • All ARIA controls must be useable by keyboard.
  • Do not use "role='presentation'" or "aria-hidden='true'" on any visible focusable element.
  • All interactive elements must have an accessible name.

ARIA Hints & Best Practices

  • Do not disable native browser focus styles
  • Keep link underlines
  • Add skip links at the top of the page in order to guide the user over areas they would like to skip
  • Don't break heading order.
  • Hide content when it isn't in focus. Use 'aria-hidden="true"' on wrapper div, everything in that div is hidden until that property is changed
  • Break your page into percievable block areas which contain semantically associated information called "regions":
    • assign landmark roles to your regions.
    • each region should have a percievable header used to label the region.
    • assign semantic navigation landmarks to the regions.
  • Be careful when you chose to auto-focus from field to field. In the event you choose to do that you must still give user firm control over when to use it b/c they might not be done with a field when the programming thinks they should be and it is a user frustration.
  • Alert your user when a max-width has been reached in an input field.
  • Do not set anything to auto-submit.
  • Ensure color is not the primary means of communication.

ARIA Keyboard Requirements

This is a requirement, not just a best practice section. We need to begin doing this.

Keyboard Bindings/Behaviors

  • We must support users who cannot use a pointing device.
  • tabindex attribute is applicable to all renderable html elements:
    • 'tab' & 'tab + shift' move focus among widgets and standard html controls
    • element with 'tabindex = 0' is added to tab sequence based on document order.
    • element with 'tabindex > 0' will be added to tab sequence based on that # value
    • element with 'tabindex < 0' will not be added to the tab sequence, but are enabled to recieve keyboard focus.
  • Once a widget has focus, arrow keys, spacebar, enter or other keys should be able to be used to navigate the options of the widget, change its state, or trigger its function(submit etc)
  • Use onfocus to track the current focus.There is no standard DOM interface to get current focus, so you'll need to use JS to keep track of it.
  • Follow keydowns to move focus. a keydown event handler can determine the next object to recieve focus and call that element's method. Use onkeydown, not onkeypress.
  • refresh the element recieving focus's style using onfocus() to avoid inconsistent browser experiences.
  • Disabled controls should not be in the tab order.
  • Dont use :focus or attribute selectors to style the focus. IE issue. Instead, set the style in an onfocus event handler.
  • Always draw focus for tabindex="-1" or any items that recieve focus IE will not always draw focus on items that are getting focus programatically.
  • Prevent used key events from performing browser functions whilst in widget. For instance, if the arrow key is pressed, prevent it from scrolling the page with some JS code.
  • If you need to have tooltips in your code (things that on hover pop up with additional info) there must be a way to activate it or access it via keyboard and with assitive technology. Use HTML title attribute as often as possible, as assistive technology will read/render it for tooltips. Currently you can also use JS if you need to to show/hide content based on focus.
                    <div class="myWidget">
                        <div class="controls">
                            <input class="widget__field" value="First Name" type="text" tabindex="0" />
                            <input class="widget__field" value="Last Name" type="text" tabindex="-1" />
                            <input type="submit" tabindex="-1" />
                        </div>
                    </div>

Roving focus example above. If you tab away and then tab back to the widget, same last active descendant becomes active again. Requires some JS to work in conjunction with the

ARIA Landmark Role Summary

Role Attributes should only be placed on an HTML5 equivalant element (nav gets a role of navigation etc) or non-semantic elements b/c they do not have semantic meaning. If you place a role on a semantic element (like a ul) most screen readers will override the role with the element's native one with the sepecified one.

banner

HTML5 Equivalant to <header>

                    <div role="banner">
                        <div class="alert">
                            <h1 class="alert__heading">
                               Warning!
                            </h1>
                            <section class="alert__message">
                                At 5:30 EST, severe weather will be incoming!
                            </section>
                        </div>
                    </div>

Aria Banner Role is for site-wide sections of content, and to be added to either a header or div, it can be added to other elements but semantics and best practice advice against. There should only be one banner role per page of content. Header elements are not considered landmarks when they are contained within a section or article element.

navigation

HTML5 Equivalant to <nav>

                    <nav class="primaryNav" role="navigation">
                        <div class="nav__list" role="">
                            <span role=""><a class="nav__item" href="">Logout</a></span>
                            <span role=""><a class="nav__item" href="">Home</a></span>
                            <span role=""><a class="nav__item" href="">About</a></span>
                            <span role=""><a class="nav__item" href="">Contact</a></span>
                            <span role=""><a class="nav__item" href="">Locations</a></span>
                        </div>
                    </nav>

One caveat with using a list to structure your nav, using lists makes it more difficult for people using screen readers as it is not conducive to the logic for it. It has been suggested to replace unordered lists with spans and divs, as these are invisible to screen readers and therefore easier to navigate. Use in conjunction with navigation role.

there can be more than one role of navigation per page. In that case, use a label or lablledby to denote separation.

complementary

HTML5 Equivalant to <aside>

                    <section class="support" role="complementary">
                        <header class="support__title">Bears lifespan</header>
                        <p class="support__content">
                            Bears live a long time. They can live without food or water for many months, subsisting only on their own stores of fat and cholesterol. Oddly enough, even in hibernation a bear can rebuild bones and avoid bone loss.
                        </p>
                    </section>

Complementary roles are the equvilant of using an aside. Think of them as tertiary content just the same as an aside as you add roles to your code.

contentinfo

HTML5 Equivalant to <footer>

                    <footer class="footer" role="contentinfo">
                        <header class="footer__title">Bears lifespan</header>
                        <p class="footer__content" role="">
                            © copyright is awesome.
                        </p>
                    </footer>

The contentinfo role is usually for the footer of the page. Again, one per page. Footer elements are not considered landmarks when they are contained within a section or article element.

form

HTML5 Equivalant to <form>

                    <form class="form" role="form">
                        <label for="firstName" class="footer__title">Bears lifespan</label>
                        <input type="text" id="firstName" role="" />
                    </form>

The form role contains form elements which can be submitted by user. Please note that this is considered a navigable landmark only when labelled with aria-label or aria-labelledby.

log

HTML5 Equivalant to <form>

                    <form class="form" role="log">
                    </form>

Region where as new info is appended, old info may dissapear (like a chat log).

main

HTML5 Equivalant to <main>

                    <main class="primaryContent" role="main">
                        <section class="bit">
                            <header class="bit__title">
                                Bears lifespan
                            </header>
                            <p class="bit__content">
                                Bears are pretty awesome, guys.
                        </section>
                    </main>

The main role is for the main content of the page. Therer should only be one main landmark per page.

region

HTML5 Equivalant to <section>

                    <div class="support" role="region">
                    </div>

NOTE THIS IS NOT AN ARIA LANDMARK

This is also a generic landmark. It should only be used when a more specific role is not available. Assistive technologies should treat this role as a complimentary region. Please note that this is considered a navigable landmark only when labelled with aria-label or aria-labelledby.

article

HTML5 Equivalant to <article>

                    <article class="support" role="article">
                    </article>

NOTE THIS IS NOT AN ARIA LANDMARK

search

HTML5 Equivalant to <form>

                    <main class="primaryContent" role="search">
                        <label for="search">Search Here</label>
                        <input type="text" id="search" role="" />
                    </form>

Reserved for the search tool.

status

HTML5 Equivalant to <form>

                    <form class="form" role="status">
                    </form>

Container for advisory info or feedback.

application

HTML5 Equivalant to <div>

                    <div class="primaryContent" role="application">
                    </div>

Represents a unique software unit. Keyboard commands handled by application rather than browser itself. Use sparingly if at all. The times it is necessary to be used will be few and far between for the most part. Please note that this is considered a navigable landmark only when labelled with aria-label or aria-labelledby.

presentation

                    <ul role="presentation">
                        <li>Item 1</li>
                        <li>Item 2</li>
                    </ul>

This removes any semantic meaning from the element to which it is assigned. This also applies to elements' children that are required (lists or tables) the child items will inherit the role from the parent.

using this label tells assistive technology that the element is semantically neutral and irrelevant for accessibility.

Do not use this if the element is focusable.

Do not use this if it has any ARIA attribute other than hidden.

Do not use if the element is referenced by another elements ARIA controls or properties.

Do not use if a required child element has been explicitly marked with a role (ex role="button") the child will have a role of 'button' and not 'presentation'

Global ARIA States/Properties

These can be set on any element regardless of whether a role is applied or not. These are supported by all roles and base markup elements.

aria-atomic

This indicates wether or not AT will present all or parts of the changed region based on change notifications from the 'aria-relevant' attribute.

The default value is 'false' if it is not set.

aria-busy (state)

This is a Live Region attribute.

This indicates wether or not an element and its subtree are being updated.

The default value is 'false' if it is not set.

Example, an author could put aria-busy to true on a page in order to show that it has not been completely ready.

                    <div role="banner">
                        <div class="alert">
                            <h1 class="alert__heading">
                               Warning!
                            </h1>
                            <section class="alert__message">
                                At 5:30 EST, severe weather will be incoming!
                            </section>
                        </div>

aria-controls (property)

This identifies the element(s) whose contents or presence is controlled by the current element.

For instance, a checkbox group might be responsible for content displaying in a different area of the site, or a tab controlling the display of its associated panel.

                    <div>
                        <button aria-expanded="false" aria-contols="tab1"/>
                    </div>
                    <section id="tab1" aria-hidden="true">
                        <p>Stop that rhyming now, I mean it!</p>
                        <p>Anybody want a peanut?</p>
                    </section>

This will work with a JS function that just toggles the aria-hidden and expanded states. Boom!

aria-describedby (property)

This identifies the element(s) that describe the object. Similar to aria-labelledby, but described is built to be more verbose. Should be a complete description.

                    <div>
                        <button />
                    </div>
                    <section >
                        <p>Are those rocks up ahead?</p>
                        <p>If they are, we all be dead!</p>
                    </section>

aria-disabled (state)

This lets AT know that the element is there but not editable or operable. A radio group might have all irrelevant options disabled, and have no focus in the tab order b/c of it. On top of using this attribute, the physical appreance should change as well (option should be greyed out etc).

                    <div>
                        <button aria-expanded="false" aria-contols="tab1"/>
                    </div>
                    <section id="tab1" aria-hidden="true">
                        <p>Stop that rhyming now, I mean it!</p>
                        <p>Anybody want a peanut?</p>
                    </section>

This will work with a JS function that just toggles the aria-hidden and expanded states. Boom!

aria-label

A string value that labels the current element with a recognizable object name.

Use aria-label if it is for text that is NOT visible on a screen. (if there is visible text labelling the element, use aria-labelledby instead)

                    <div class="control">
                        <div class="controls">
                           <button aria-label="Close" onclick="myAction.close()">X</button>
                        </div>
                    </div>

Because there is nothing in the button text that actually says it closes the page, using a label there is a good practice.

aria-labelledby

This represents a 'relationship' attribute, meaning it defines association between elements.

ID's the element that is labelling the current element. Same purpose as aria-label; to give the user a recognizable object name. Note that there are 3 'l's total in that, not 2.

If the text is visible on screen, use aria-labelledby. Screen readers give precedence to aria-lablledby over aria-label.

By default, assistive technologies do not relay hidden information (stuff hidden by css display none or visibility hidden) but that can be overridden by the author if they put aria-labelledby or aria-describedby on something that is visually hidden.

                    <label>
                        Name
                        <input type="text" aria-labelledby="error-message">
                    </label>
                    <span id="error-message" style="display:none; aria-hidden="true"/>
                        OH SNAP YOU GOTS ERR-OHRS.
                    </span>

This code represents a hidden field used as the label for an input. Note it is still referenced even though it is aria-hidden='true' and is display:none; via css. Boom. 

                    <div id="action" class="action">
                        <div>
                            <div id="name">Name</div>
                            <input type="text" aria-labelledby="name action"/>
                        </div>
                        <div>
                            <div id="phone">Phone</div>
                            <input type="text" aria-labelledby="phone action"/>
                        </div>
                    </div>

This code represents input fields being labelled by both inidividual labels and a group label on an ancestor.

aria-hidden (state)

This indicates that this element and all its descendants are not visible or perceivable to any user.

If that element is only visible after a user action, the attribute has to be set to 'true'. When that element is visible, the attribute must be changed from 'true' to 'false' in order to indicate the state change. This must be set regardless of the mechanic used to hide/show the item. This allows hidden elements to be skipped by assistive technologies/user agents. CSS attribute selector can be used to hide a hidden element, while JS would only be used to update the value of the attribute on click.

You can use this to hide visibly rendered content from assistive technologies ONLY if the act of hiding the content is intended to improve the experience for assistive technology users by removing redundant or extraneous content. If you do that, you must ensure that identical or equivalent meaning/functionality is exposed to assistive technology. This is a real liability though, so don't do this unless there is no other alternative.

Both CSS declarations "visibility: hidden;" and "display: none;" hide the content from assistive technologies that access the DOM through rendering engine.

                    <div aria-hidden="false">
                        <div class="alert">
                            This message is visible!!
                        </div>
                    </div>

Aria hidden!

aria-required (state)

Indicates a required field. Can be used in conjunction with aria-invalid.

When user attempts to submit wrong data on a field with this attribute, the author can use the aria-invalid attribute to signal an error. However this must run AFTER a user has clicked 'submit' on a widget, b/c the user has not actually entered data up to that point.
